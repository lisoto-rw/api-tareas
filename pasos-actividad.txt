    Qué problemas aparecen cuando la integración y el despliegue se realizan manualmente.
    
    cosas buenas sinergias entre desarrollo y operaciones, control de flujo de cambio de parte de desarrolladores a operaciones.

    El "en mi máquina funciona": Que el código ande en la computadora de un compañero no asegura que ande en la tuya o en el servidor.
    Errores humanos: Te podés olvidar de copiar un archivo, errarle a un comando o subir una versión vieja por accidente.
    Pérdida de tiempo: En vez de programar, el equipo pierde horas configurando, instalando dependencias y subiendo archivos pesados a mano.
    Miedo a desplegar: Como es difícil y estresante, se hace pocas veces al mes, acumulando demasiados cambios juntos (y si algo falla, buscar el error es una pesadilla).



    Qué rol cumplen la colaboración, la automatización y el feedback en DevOps.

    Colaboración: Rompe la pared entre los que programan (Dev) y los que mantienen los servidores (Ops). Todos tiran para el mismo lado y comparten la responsabilidad del proyecto.
    Automatización: Es dejar que las máquinas hagan el trabajo aburrido y repetitivo (probar el código, armar la imagen de Docker, subirla). Esto evita errores y ahorra tiempo.
    Feedback (Devolución rápida): Es saber ya mismo si lo que hiciste está bien o mal. Si rompiste algo, el sistema te avisa en minutos para que lo arregles enseguida, antes de que llegue al usuario.

    si hay una falla en el pipeline, no se generan las imagenes correspondientes, que seguian almacenadas en el contenedor del registro para luego descargar y desplegar, no se generan entonces no se puede poner en produccion o testing nada o en algun otro ambito.


4. ¿Qué sucede cuando una etapa del pipeline falla?

El proceso se frena de inmediato: Si los tests fallan o la imagen de Docker no se puede armar, el pipeline se detiene como un botón de emergencia de una fábrica. No se pasa a la siguiente etapa.
    Alerta roja: GitHub Actions les va a mostrar una cruz roja gigante y (por lo general) manda un mail avisando qué falló.
    Seguridad: El entorno de testing (o el servidor real) queda a salvo. El código roto nunca llega a desplegarse, asegurando que la aplicación que ven los usuarios siga funcionando perfectamente con la versión anterior.

Para la guia pide hacer una modificacion de la estructura del proyecto

En este momento, en tu archivo app.js tenés mezcladas dos responsabilidades distintas:

Definir la API: Configurar las rutas (/tareas), usar express.json(), etc.

Encender el servidor: El método app.listen(PORT, ...) que hace que tu compu se ponga a escuchar en el puerto 3000.


¿Por qué te piden separarlo? (El motivo con Jest, Supertest y Docker)
Para Supertest: Supertest es una librería que sirve para probar tus rutas simulando peticiones HTTP (como si usaras Postman, pero en código). Para hacer esto de manera eficiente, Supertest necesita la configuración de Express (app), pero NO necesita que el servidor esté prendido escuchando en un puerto. Si dejas el app.listen() ahí, cada vez que corras los tests de Jest, la aplicación va a intentar levantar el servidor real en el puerto 3000, lo que puede dar errores de "puerto ya ocupado" o dejar los procesos colgados.

Para Docker: Al separar el arranque de la lógica, tu contenedor de Docker va a llamar exclusivamente al archivo que enciende el servidor (server.js), mientras que Jest solo va a llamar al archivo que tiene la lógica (app.js).

La solución es dividir tu app.js actual en dos archivos: 
uno se va a seguir llamando app.js (solo lógica) y 
vamos a crear uno nuevo llamado server.js (solo encendido).

ajuste de packages.json

{
  "scripts": {
    "start": "node server.js"
  }
}
